Functions

Functions can easily get complicated with the amount of statements in them. Along with poor naming conventions, larger functions may take plenty of time to fully understand at all. When examining functions, try to look out for the following points.

Size

Preferably, functions should be small. A small function is simple to read and does one thing. Even when you think a function is small, it can probably be smaller. To make functions smaller, extract complex chains and nested statements into new functions. It is easy to get carried away by writing code in a single function. And that is okay, as long as you refactor parts into other functions.

Complexity

Function complexity can be measured by the number of blocks that exist. Every if statement and loop generates added complexity. Creating nested structures can multiply the complexity.

Any side effects included should also be a red flag. A common mistake is writing a function that checks if a condition is true or false. This function may make calls before returning the value. By doing this, we are adding responsibilities that should remain by the caller.

The job of a function is to perform an action or answer a question that the caller is asking. When a function tries to do both of these jobs, there is confusion to which one the function is attempting to do. An example is having a method that changes a value and returns true or false depending on if the operation was successful. A better way to do the same is to have two functions, one that checks if the operation will be successful and another that does the operation itself. In that way you only perform the operation if it is successful.

Arguments

A function's arguments can be as detrimental to its complexity as the other points. Arguments inevitably indicate dependencies that exist within the function.

Ideally, functions should have no arguments at all, if possible. The reasoning is that testing becomes easier with less arguments to choose from. When the number of arguments increases past three or more, the combination of arguments can be staggering. It is much easier to understand a function that does less and is not dependent on other values.

The one argument that should always be avoided is a boolean. A boolean immediately indicates that a function may do two or more things. The caller of the function would also know what operation to perform in the first place.
