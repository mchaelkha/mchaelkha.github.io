Clean Code

“You are reading this book for two reasons. First, you are a programmer. Second, you want to be a better programmer. Good. We need better programmers.”

Robert C. Martin - Clean Code: A Handbook of Agile Software Craftsmanship

Preface

I decided to base my project on a book that I recently started reading. Clean Code was first published in 2008. Since then, it has aged a fair bit. I definitely recommend reading sections of the book that seem interesting to you. The book still sends a common message about best practices for writing readable, maintainable code. This site hopes to capture the main points with some of my own thoughts and examples from a contemporary standpoint.

Writing code can arguably be the same as writing an essay. There are conventions to follow so that it is readable by an audience. You do not want your audience to struggle over a single sentence. There is a logical flow to a story much like how code should communicate.

Why?

Reading code is as important as writing code. Reading code can be challenging however. We all have our own individual coding style and conventions. We may know how to program in a programming language, but code in the same language cannot always be universally interpreted due to styling.

I like to compare this to how we understand the English language. We know what words exactly mean since they are well-defined. Words can be synonymous, such as little and small. Parts of a sentence can be moved around. Take the following two sentences:

  • Jogging is one of his favorite daily activities.
  • One of his favorite daily activities is jogging.

Ultimately the two sentences have the same meaning. There is a slight difference however. This subtlety goes into the difference between active and passive voice. You may wish to use the active voice to emphasize the noun, 'jogging', while the passive voice would be used to focus on the action, 'daily activities'.

Code can be very similar in nature. For instance, languages often support multiple ways of performing the same operations through syntactic sugar solely for the purpose of reading and expressing. A common example is the usage of assignment operators. It is a shorthand way of performing operations and assigning on a value. In most modern languages, it is actually memory-efficient use this method over the traditional way of writing the operators separately.

Reading the same code written in other languages should also be universal. There may be syntactic differences between C#, C++, and Java, which are all very alike. The main differences between these languages is the naming of types and functions. You can make similar connections to other languages.

Define It

There are many interpretations as to what clean code should entail. The common elements define clean code as straightforward, easy to maintain, fully optimized, has unit and acceptance tests, minimizes dependencies, and has meaningful names. All in all, most developers can agree that code should be readable by others.

There should be an emphasis on authorship. When writing code, you are providing readers an opportunity to understand. As an author you are communicating the purpose of your work. In fact, you should consider how much code you read compared to the amount of time spent writing.
